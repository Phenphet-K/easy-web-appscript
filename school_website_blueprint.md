# 🏫 พิมพ์เขียวสถาปัตยกรรมระบบเว็บไซต์โรงเรียน (Google Apps Script Dynamic CMS)
เอกสารฉบับนี้อธิบายถึงสถาปัตยกรรมระบบ โครงสร้างฐานข้อมูล โฟลว์การทำงาน และตรรกะทางเทคนิค (Logic) ทั้งหมดของระบบเว็บไซต์โรงเรียน เพื่อใช้ส่งมอบให้โปรแกรมเมอร์นำไปเขียนโค้ดต่อได้ทันที โดยไม่มีการใส่โค้ดดิ้งเต็มรูปแบบ แต่จะเน้นอธิบายวิธีคิดและวิธีการทำงานของแต่ละส่วนอย่างละเอียด

---

## 1. ภาพรวมสถาปัตยกรรมระบบ (System Architecture)
ระบบนี้ถูกออกแบบมาในรูปแบบ **Serverless Web Application** โดยใช้ทรัพยากรฟรีของ Google ทั้งหมด:
*   **Frontend (หน้าบ้าน):** พัฒนาเป็น Single Page Application (SPA) ด้วย HTML5, CSS3 (Vanilla CSS) และ Javascript (ES6) ทำหน้าที่สลับหน้าเพจ ดึงข้อมูลมาแสดงผล และส่งคำสั่ง CRUD ไปยังหลังบ้าน
*   **Backend (หลังบ้าน):** รันบน Google Apps Script (`Code.gs`) ทำหน้าที่ควบคุมการเข้าถึงฐานข้อมูล, ระบบความปลอดภัย (Session), การจัดการไฟล์รูปภาพ และแปลงลิงก์ข้อมูลแผนที่
*   **Database (ฐานข้อมูล):** ใช้ Google Sheets เป็นฐานข้อมูลแบบ Relational Database จำลอง
*   **File Storage:** ใช้ Google Drive เป็นพื้นที่เก็บไฟล์ภาพกิจกรรมและโลโก้โรงเรียน

```
[ Frontend: SPA (HTML/JS) ] 
       │ ▲
       │ │  เรียกใช้ฟังก์ชันฝั่ง Server (google.script.run)
       ▼ │
[ Backend: Google Apps Script (Code.gs) ]
       ├── [ API แผนที่ ] (ดึงพิกัดจากลิงก์ Redirect)
       ├── [ Google Sheets ] (ฐานข้อมูลจัดเก็บข้อความ/การตั้งค่า)
       └── [ Google Drive ] (พื้นที่จัดเก็บไฟล์รูปภาพ)
```

## 2. โครงสร้างฐานข้อมูล (Database Schema)
ใน Google Sheets ให้สร้างแผ่นงาน (Sheet) ทั้งหมด 6 ตารางหลัก ดังนี้:

### 2.1 ตาราง `Settings` (เก็บการตั้งค่าเว็บไซต์และการตกแต่ง)
ทำหน้าที่เก็บคู่ตัวแปร (Key-Value) สำหรับนำไปเรนเดอร์ธีม ดีไซน์ และข้อมูลพื้นฐานของโรงเรียน
*   **โครงสร้างคอลัมน์:** `key` | `value`
*   **ตัวแปรสำคัญที่ต้องมี:**
    *   ข้อมูลพื้นฐาน: `schoolName`, `siteTitle` (หัวข้อแท็บเบราว์เซอร์), `schoolSubtitle`, `schoolAddress`, `contactPhone`, `contactEmail`, `facebookUrl`, `lineOaUrl`
    *   ภาพตกแต่ง: `logoUrl`, `heroImageUrl`
    *   ธีมสี (ใช้รูปแบบ Hex Code): `themeColorPrimary` (สีหลัก), `themeColorSecondary` (สีรอง), `themeColorGold` (สีเน้น/ทอง), `themeBackgroundColor` (สีพื้นหลัง)
    *   ฟอนต์และดีไซน์: `themeFontFamily` (ตระกูลฟอนต์), `themeFontSizeBase` (ขนาดตัวอักษรพื้นฐาน), `themeFontSizeHeading` (ขนาดหัวข้อ)
    *   สารจากผู้อำนวยการ (แสดงหน้าแรก): `aboutDirectorName`, `aboutDirectorRole`, `aboutDirectorMessage`, `aboutDirectorImageUrl`
    *   ประวัติ วิสัยทัศน์ พันธกิจ: `aboutHistory`, `aboutVision`, `aboutMission`
    *   แผนที่ตั้ง: `contactMapEmbedUrl` (เก็บลิงก์ฝังแผนที่ที่ผ่านการแปลงแล้ว)

### 2.2 ตาราง `News` (เก็บข่าวประชาสัมพันธ์และกิจกรรมล่าสุด)
*   **โครงสร้างคอลัมน์:** `id` | `title` | `content` | `imageUrl` | `category` | `author` | `date` | `published`
*   **ข้อกำหนดทางเทคนิค:**
    *   `id`: รหัสประจำข่าวสาร (แนะนำใช้ระบบ UUID เพื่อความปลอดภัยและไม่ซ้ำกัน)
    *   `imageUrl`: จัดเก็บแบบ String คั่นด้วยเครื่องหมายจุลภาค (Comma-Separated) เพื่อให้หนึ่งข่าวรองรับการบันทึกภาพประกอบได้หลายรูป (Multi-Image)
    *   `date`: จัดเก็บวันเวลาที่เขียนข่าวด้วยรูปแบบ ISO String เสมอ
    *   `published`: เป็นค่าสถานะเปิด/ปิดการเผยแพร่ข่าว (TRUE/FALSE)

### 2.3 ตาราง `Activities` (คลังภาพกิจกรรมโรงเรียน)
*   **โครงสร้างคอลัมน์:** `id` | `title` | `description` | `imageUrl` | `date` | `published`
*   **ข้อกำหนดทางเทคนิค:**
    *   `imageUrl`: รองรับรูปภาพหลายรูปเช่นเดียวกับตารางข่าวสาร (เก็บเป็นลิงก์คั่นด้วยเครื่องหมายจุลภาค)

### 2.4 ตาราง `Events` (ปฏิทินกิจกรรมโรงเรียน)
*   **โครงสร้างคอลัมน์:** `id` | `title` | `description` | `eventDate` | `time` | `location` | `color` | `published`
*   **ข้อกำหนดทางเทคนิค:**
    *   `eventDate`: บันทึกวันที่จัดกิจกรรมในรูปแบบ YYYY-MM-DD
    *   `color`: ระบุสีของแท็กกิจกรรมบนหน้าปฏิทิน (เช่น blue, red, green, gold)

### 2.5 ตาราง `Board` (ทำเนียบผู้บริหารและบุคลากร)
*   **โครงสร้างคอลัมน์:** `id` | `name` | `role` | `imageUrl` | `published` | `order`
*   **ข้อกำหนดทางเทคนิค:**
    *   `order`: จัดเก็บตัวเลขลำดับ (Integer) เพื่อนำไปจัดระดับขั้นและเรียงลำดับการแสดงผลผู้บริหารบนหน้าเว็บ (เช่น ลำดับ 1 คือ ผู้อำนวยการ, ลำดับ 2-3 คือ รองผู้อำนวยการ) เพื่อความยืดหยุ่นในการปรับตำแหน่งหน้าที่ของบุคลากรได้อย่างละเอียดในอนาคต

### 2.6 ตาราง `Admins` (ข้อมูลบัญชีผู้ดูแลระบบ)
*   **โครงสร้างคอลัมน์:** `username` | `password_hash` | `name` | `created_at`
*   **ข้อกำหนดทางเทคนิค:**
    *   `password_hash`: เก็บรหัสผ่านที่ผ่านการแปลงเป็น Hash ด้วยอัลกอริทึม SHA-256 เพื่อความปลอดภัยขั้นสูง (ห้ามบันทึกเป็นรหัสผ่านตรง ๆ)

### 2.7 ตาราง `Updates` (เก็บข้อมูลข่าวประกาศอัปเดตโรงเรียน)
*   **โครงสร้างคอลัมน์:** `id` | `title` | `content` | `date` | `published`
*   **ข้อกำหนดทางเทคนิค:**
    *   `content`: เก็บบันทึกข้อมูลเนื้อหาข่าวประกาศอัปเดตในรูปแบบ HTML เพื่อให้คงสไตล์การตกแต่งที่ออกแบบจากโปรแกรม Rich Text Editor ไว้ได้
    *   `published`: เป็นค่าสถานะเปิด/ปิดการเผยแพร่ประกาศ (TRUE/FALSE)

## 3. ขั้นตอนวิธีการทำงานฝั่ง Server (Backend: Code.gs)

โปรแกรมเมอร์ต้องพัฒนาส่วนหลังบ้านให้มีระบบหลัก ๆ ดังนี้:

### 3.1 ระบบจัดการตารางอัตโนมัติ (Auto-Initialization & Migration)
*   เมื่อมีการเข้าหน้าเว็บ ตัวเซิร์ฟเวอร์ต้องตรวจสอบว่าแผ่นงาน (Sheets) ทั้งหมดที่ระบุในข้อ 2 ครบถ้วนหรือไม่ หากไม่ครบให้สร้างแผ่นงานที่ขาดหายไปขึ้นมาทันที
*   **ระบบย้ายข้อมูลคณะผู้บริหาร (Board Migration):** หากเป็นระบบเวอร์ชันเก่าที่ยังใช้โครงสร้างเก็บชื่อผู้บริหารในตาราง `Settings` (คีย์ที่ขึ้นต้นด้วย `aboutDirector` และ `aboutDeputy`) เมื่อสร้างตาราง `Board` ใหม่ขึ้นมา หลังบ้านต้องดำเนินการย้ายข้อมูลชื่อ ตำแหน่ง และรูปภาพของผู้บริหารเดิมเข้าไปยังตาราง `Board` ใหม่ให้อัตโนมัติ พร้อมตั้งค่าลำดับความสำคัญ (`order`) 1, 2, และ 3 ตามลำดับ เพื่อป้องกันข้อมูลสูญหายโดยที่ผู้ใช้งานไม่ต้องเริ่มตั้งค่าใหม่

### 3.2 ระบบสิทธิ์ผู้ดูแลระบบ (Authentication & Session Manager)
*   **ระบบ Login:** ตรวจสอบผู้ใช้และรหัสผ่าน โดยการนำรหัสผ่านที่ส่งเข้ามาไปแฮช SHA-256 ก่อนนำไปเปรียบเทียบกับตาราง `Admins`
*   **Session Token:** เมื่อเข้าสู่ระบบสำเร็จ ให้เซิร์ฟเวอร์สร้างรหัสสิทธิ์สุ่มขึ้นมา (UUID) และเก็บไว้ในระบบ `ScriptProperties` พร้อมกำหนดอายุการใช้งาน (เช่น 8 ชั่วโมง)
*   **Session Validation:** ทุกฟังก์ชันที่ทำหน้าที่ เพิ่ม แก้ไข หรือลบข้อมูล (CRUD) จะต้องส่ง Token นี้เข้ามาตรวจสอบกับฝั่ง Server ก่อนเริ่มทำงานเสมอ หากหมดอายุหรือไม่มีสิทธิ์ ให้ปฏิเสธการเข้าถึง

### 3.3 การดึงข้อมูลแบบปลอดภัย (Safe JSON Serialization Safeguard)
> [!IMPORTANT]
> **ปัญหาสำคัญในการพัฒนาเว็บด้วย Google Apps Script:** หากดึงข้อมูลดิบในแถวตารางที่มีคอลัมน์ประเภท Date (เช่น วันที่จัดกิจกรรม) หรือ Object อื่น ๆ ส่งผ่านคำสั่ง `google.script.run` ไปให้หน้าบ้าน ตัว Apps Script จะไม่สามารถแปลงประเภทข้อมูลที่เป็น Date ส่งกลับไปได้ ทำให้เกิดข้อผิดพลาดเงียบและส่งข้อมูลกลับเป็นค่าว่างเปล่า (`null`) ส่งผลให้หน้าเว็บค้างที่สถานะ "กำลังโหลด..."
*   **วิธีการทำงานแก้ไข:** หลังบ้านต้องมีฟังก์ชันสุขอนามัยข้อมูล (Sanitization Helper) ทำหน้าที่ประมวลผลข้อมูลแบบวนลูป (Recursively) เพื่อตรวจสอบประเภทตัวแปรทุกตัวในชุดข้อมูล หากพบข้อมูลประเภท Date ให้แปลงเป็น ISO String หรือจัดรูปแบบให้เป็นประเภท String ธรรมดาก่อนส่งเข้าสู่ขั้นตอนแปลงเป็น JSON เสมอ

### 3.4 ระบบอัปโหลดรูปภาพผ่าน Google Drive
*   รับข้อมูลรูปภาพที่อัปโหลดมาในรูปแบบเข้ารหัสอักขระ Base64 จากหน้าบ้าน
*   **การจัดโครงสร้างโฟลเดอร์แบบไดนามิก (Dynamic Folder Structure):** เพื่อป้องกันไม่ให้โฟลเดอร์ Google Drive หลักมีจำนวนไฟล์รูปภาพอัดแน่นกันมากเกินไป หลังบ้านต้องตรวจสอบและสร้างโฟลเดอร์ย่อย (Subfolder) ภายใต้ไดเรกทอรีจัดเก็บภาพหลัก โดยตั้งชื่อตามวันที่ปัจจุบันในรูปแบบปี-เดือน-วัน (YYYY-MM-DD) เพื่อจัดรูปภาพที่อัปโหลดเข้าแยกเก็บตามวันใช้งาน
*   กำหนดสิทธิ์เข้าถึงโฟลเดอร์ย่อยและไฟล์ภาพให้เป็นแบบ "ทุกคนที่มีลิงก์สามารถดูได้" (Anyone with link can view)
*   ส่งที่อยู่ภาพ (URL) กลับไปในรูปแบบลิงก์ภาพขนาดย่อที่มีประสิทธิภาพ:
    `https://drive.google.com/thumbnail?id=[FILE_ID]&sz=w1200` (วิธีนี้จะโหลดภาพเร็วขึ้นและช่วยให้เว็บไซต์ลื่นไหล)

### 3.5 ระบบแยกและแปลงลิงก์ Google Maps อัตโนมัติ (Google Maps Redirect Resolver)
เมื่อแอดมินแก้ไขลิงก์แผนที่ของโรงเรียน ให้ประมวลผลก่อนจัดเก็บดังนี้:
1.  **ดึงที่อยู่ต้นทาง:** หากอินพุตมีแท็ก iframe ให้สแกนดึงเฉพาะตัวลิงก์ข้างในคุณลักษณะ `src`
2.  **จัดการลิงก์สั้น (`maps.app.goo.gl` หรือ `goo.gl/maps`):**
    *   เนื่องจากเบราว์เซอร์ไม่สามารถเข้าถึงลิงก์สั้นผ่าน Iframe ได้เนื่องจากโดนบล็อกด้านความปลอดภัย (CORS) หลังบ้านจะต้องประมวลผลแทน
    *   ใช้คำสั่งเรียกไปยัง URL ต้นทางด้วยตรรกะบล็อกการติดตามแบบ Redirect (`followRedirects: false`)
    *   อ่านค่าจากหัวข้อการตอบกลับ (Response Header) ในส่วน `Location` เพื่อแกะหาลิงก์ปลายทางที่สมบูรณ์
3.  **ประมวลผลลิงก์ยาว:**
    *   วิเคราะห์ลิงก์ปลายทางด้วยรูปแบบข้อความ (Regex) เพื่อดึงพิกัดละติจูดและลองจิจูด (Latitude/Longitude) หรือคำค้นหา (Search Query) ออกมา
    *   สร้างลิงก์สำหรับฝัง (Embed Url) ขึ้นมาใหม่ให้ปลอดภัยและเป็นมาตรฐาน:
        `https://maps.google.com/maps?q=[Latitude],[Longitude]&z=17&output=embed`
    *   จัดเก็บลิงก์นี้ลงไปในค่าตั้งค่าแทนที่ค่าเดิม ทำให้แผนที่โรงเรียนโหลดขึ้นบนหน้าจอได้อย่างสมบูรณ์

## 4. โฟลว์การทำงานและตรรกะฝั่งหน้าบ้าน (Frontend Client-Side)

### 4.1 ตรรกะของระบบสไลด์โชว์หน้าแรก (Dynamic Hero Slideshow)
*   **ตรรกะการรวบรวมรูปภาพ (Pooling Image):** ดึงภาพแรกจากอัลบั้มภาพของข่าวสารประชาสัมพันธ์ (`News`) และภาพแกลเลอรีจากภาพกิจกรรม (`Activities`) มารวมกัน
*   **กรณีที่ไม่มีข้อมูลภาพ:** ให้ใช้รูปภาพแบนเนอร์ของโรงเรียน (`heroImageUrl` จากการตั้งค่า) เป็นรูปพื้นหลังหลักเริ่มต้น (Fallback Banner)
*   **การควบคุมภาพเคลื่อนไหว:** มีเวลาหน่วงในการเปลี่ยนเฟรมโดยอัตโนมัติทุก ๆ 5 วินาที พร้อมปุ่มควบคุมเปลี่ยนรูปภาพไปข้างหน้า/ย้อนกลับ และตัวระบุตำแหน่งสไลด์ (Dot Indicators) ที่โต้ตอบได้จริง

### 4.2 ตรรกะของปฏิทินกิจกรรมโรงเรียน (Monthly Event Calendar)
*   **ตรรกะการเรนเดอร์ปฏิทินประจำเดือน:**
    *   ดึงข้อมูลเดือนและปีที่ต้องการแสดงผล คำนวณวันแรกของสัปดาห์ (วันอาทิตย์-วันเสาร์) และจำนวนวันทั้งหมดในเดือนนั้น ๆ เพื่อวาดตาราง และแสดงผลเดือน/ปีในรูปแบบพุทธศักราช (ปี ค.ศ. + 543)
    *   **การโฟกัสวันปัจจุบัน:** ตรวจสอบหา "วันปัจจุบัน" และแสดงผลด้วยการเพิ่มกรอบไฮไลท์สี่เหลี่ยมรอบวันนั้น ๆ โดยใช้คุณสมบัติ CSS Box Shadow เพื่อสร้างเส้นขอบโฟกัส (Focus Outline) ที่ชัดเจน
    *   **การระบุกิจกรรม:** จับคู่ข้อมูลวันที่แต่ละช่องกับตาราง `Events` ฝั่งฐานข้อมูลเพื่อหาว่าวันไหนมีกิจกรรมบ้าง หากมีให้แสดงผลพื้นหลังของช่องวันนั้น ๆ ด้วยสีประจำหมวดหมู่ของกิจกรรมแบบเต็มกรอบสี่เหลี่ยม (Full-colored cell backgrounds) เช่น สีฟ้า, สีแดง, สีเขียว, สีทอง เพื่อสร้างจุดเด่นสะดุดตา โดยช่องที่มีกิจกรรมนี้จะไม่มีการแสดงกรอบโฟกัสรอบช่อง ยกเว้นจะเป็นวันปัจจุบันที่มีกิจกรรมในตัว
*   **ระบบกรองและค้นหากิจกรรม (Date Filtering & Selection):**
    *   เมื่อผู้เข้าชมคลิกเลือกวันที่ใด ๆ บนหน้าปฏิทิน หน้าบ้านจะกรองและแสดงเฉพาะรายการกิจกรรมที่ตรงกับวันนั้น ๆ ที่ตารางแสดงกิจกรรมด้านข้าง
    *   หากคลิกซ้ำที่วันเดิม (การยกเลิกเลือก) ให้ระบบรีเซ็ตและกลับมาแสดงกิจกรรมทั้งหมดที่กำลังจะเกิดขึ้นนับจากวันปัจจุบันเป็นต้นไป

### 4.3 ตรรกะการแสดงผลทำเนียบผู้บริหารแบบไดนามิก (Dynamic Board Rendering)
*   หน้าบ้านอ่านข้อมูลรายชื่อผู้บริหารทั้งหมดที่ดึงมาจากฐานข้อมูลชีต `Board` ซึ่งเรียงตามคอลัมน์ `order` จากน้อยไปหามากมาแล้ว
*   สร้างกล่องข้อมูลผู้บริหาร (Card) ขึ้นมาแสดงผลตามจำนวนจริง
*   **กฎการแสดงผลคนแรก (First-Child Custom Styling):** ผู้บริหารคนแรกสุด (เช่น ตำแหน่งผู้อำนวยการโรงเรียน) จะถูกเรนเดอร์ในสไตล์การ์ดที่มีขนาดใหญ่กว่าคนอื่น (เช่น 320px แทนที่จะเป็น 280px) พร้อมพื้นหลังไล่ระดับและกรอบสีทองพิเศษโดยอัตโนมัติ เพื่อสร้างลำดับความสำคัญตามตำแหน่งทางราชการอย่างโดดเด่น

### 4.4 ตรรกะความปลอดภัยสองชั้นสำหรับการฝังแผนที่โรงเรียน
*   นอกจากจะจัดการแปลงพิกัดและถอดรหัสในระบบหลังบ้านแล้ว ในฝั่งหน้าบ้าน ในขั้นตอนที่รับข้อมูลและตั้งค่าลิงก์แผนที่ไปแสดงบน Iframe ให้เขียนฟังก์ชันตัวแปลงแบบ Regex ควบคู่ไปด้วย
*   หากแอดมินหรือโปรแกรมเมอร์แอบเข้าไปแก้ไขค่าแผนที่ในชีต Settings โดยตรง หน้าบ้านจะช่วยกรองและแปลงลิงก์ Google Maps ที่ใส่เข้ามาเป็นรูปแบบลิงก์สำหรับฝัง (`output=embed`) ให้อัตโนมัติอีกหนึ่งชั้น เพื่อป้องกันปัญหาเฟรมแผนที่ไม่แสดงผล (Refused to display)
*   **ระบบ Fallback สำหรับลิงก์สั้นที่ถอดรหัสไม่สำเร็จ:** หากลิงก์สั้น (`maps.app.goo.gl` หรือ `goo.gl/maps`) ไม่ได้รับการแปลงพิกัดจากระบบหลังบ้านก่อนส่งมา (เช่น เนื่องจากผู้ใช้พิมพ์ข้อมูลลงสเปรดชีตตรงๆ หรือข้อจำกัดด้านเน็ตเวิร์ก) หน้าบ้านจะดึงเอาข้อมูลที่อยู่โรงเรียน (`schoolAddress`) หรือชื่อโรงเรียน (`schoolName`) มาสร้างเป็นลิงก์คำค้นหาสำหรับฝังแผนที่แสดงพิกัดทดแทนทันที เพื่อหลีกเลี่ยงข้อผิดพลาดเฟรมแผนที่ว่างเปล่า และจะแสดงลิงก์ข้อความใต้เฟรมเพื่อให้สามารถคลิกเปิดลิงก์ต้นทางไปยังแท็บใหม่ภายนอกได้


### 4.5 ฟังก์ชันถอดรหัสและแปลง URL รูปภาพ Google Drive (Google Drive Image URL Resolver)
*   เพื่อป้องกันปัญหารูปภาพไม่แสดงผลเนื่องจากแอดมินนำที่อยู่รูปภาพในลิงก์แชร์ของ Google Drive หลายรูปแบบมาใส่ในตาราง หน้าบ้านต้องติดตั้งฟังก์ชันดักจับ (Regex Pattern Matching) เพื่อแปลง URL ให้กลายเป็นลิงก์สำหรับแสดงผลเป็นรูปภาพได้โดยตรง (Direct Image Stream)
*   รองรับโครงสร้างลิงก์ Google Drive ทั้ง 3 รูปแบบหลัก ได้แก่:
    1.  `lh3.googleusercontent.com/d/[FILE_ID]`
    2.  `[URL_PARAMS]?id=[FILE_ID]`
    3.  `google.com/file/d/[FILE_ID]/view`
*   หากตรวจพบรูปแบบดังกล่าว ให้ดึงเอาเฉพาะ `FILE_ID` ของไฟล์ภาพออกมา และแปลงเป็นที่อยู่ลิงก์ขนาดย่อมาตรฐาน:
    `https://drive.google.com/thumbnail?id=[FILE_ID]&sz=w1200`
*   หากไม่ใช่ลิงก์ของ Google Drive ให้ส่งลิงก์เดิมกลับไปใช้งานโดยตรง

### 4.6 ระบบพรีวิวธีมแบบเรียลไทม์ (Live Theme Preview)
*   ในหน้าแก้ไขตั้งค่าความสวยงามของโรงเรียน เมื่อผู้ดูแลระบบคลิกเลือกแถบสี หรือเปลี่ยนรูปแบบฟอนต์และขนาดตัวอักษร
*   ตัวระบบฝั่งเบราว์เซอร์จะดักจับเหตุการณ์การเลือกตัวเลือก (Input & Change Events) และทำการปรับปรุงค่าตัวแปร CSS (CSS Custom Properties) บน Document Root ของเว็บไซต์โดยทันที
*   ตรรกะนี้ช่วยให้แอดมินเห็นความเปลี่ยนแปลงด้านโทนสีของเว็บไซต์ทั้งหมดได้ทันทีบนหน้าจอหลักในเวลาจริงแบบเรียลไทม์ เพื่อประกอบการตัดสินใจตกแต่งก่อนตัดสินใจบันทึกค่าลงไปยังฐานข้อมูลฝั่งเซิร์ฟเวอร์

### 4.7 การควบคุมการแสดงผลรูปภาพและขนาดตัวอักษรแบบ Responsive (Responsive Media & Typography Control)
*   **การควบคุมรูปภาพประชาสัมพันธ์ (News Image Rendering):** เพื่อไม่ให้ภาพหน้าปกข่าวสารและภาพกิจกรรมถูกตัดขอบ (Cropped) หรือข้อความบนภาพแบนเนอร์ขาดหาย ให้กำหนดกล่องแสดงรูปภาพข่าวสารใช้สัดส่วนแบบ `aspect-ratio: 16 / 9` และใช้คุณสมบัติ `object-fit: contain` เพื่อแสดงภาพเต็มขนาดเสมอ โดยใช้สีพื้นหลังเข้ม `var(--primary-light)` ในการเติมเต็มช่องว่างที่เหลือเพื่อความสวยงามในรูปแบบการ์ด
*   **การควบคุมรูปภาพในหน้าต่างป๊อปอัปรายละเอียด (Modal Image Frame):** กำหนดให้รูปภาพหลักที่แสดงบนโมดอลรายละเอียดประชาสัมพันธ์และภาพกิจกรรมแสดงผลเต็มรูปด้วยสัดส่วน `object-fit: contain` และมีสีพื้นหลังกรอบเป็นสีดำสนิท `#000000` โดยมีความสูงสูงสุดจำกัดที่ `450px` เพื่อให้อ่านข้อความข่าวสารหรือแบนเนอร์ประกาศบนภาพได้ครบถ้วน
*   **การปรับขนาดตัวอักษรบนหน้าจอมือถือ (Responsive Typography Scaling):** ภายใต้เงื่อนไขขนาดหน้าต่างเบราว์เซอร์ไม่เกิน `768px` ตัวสไตล์ชีท (CSS Media Queries) จะต้องควบคุมการลดขนาดของตัวอักษรหัวข้อต่าง ๆ ลงโดยใช้ฟังก์ชันคำนวณสเกลตามอัตราส่วนของขนาดหัวข้อหลัก เช่น `calc(var(--font-size-heading) * 0.78)` สำหรับ `.section-title`, `.executives-title`, และ `.modal-body h2` เพื่อป้องกันไม่ให้อักษรล้นบรรทับและจัดระเบียบตัวอักษรแบรนด์โรงเรียน `.school-name h1` ให้กระชับสวยงาม

## 5. ระบบหลังบ้านผู้ดูแลระบบ (Admin CMS Dashboard)

โปรแกรมเมอร์จะต้องพัฒนาแผงควบคุมระบบ (CMS Panel) ที่มีโฟลว์การทำงานแบบเรียลไทม์ (Reactive UI Update):

```
[ แผง Admin ทำการเพิ่ม/แก้ไข/ลบข้อมูล ] ── ส่งข้อมูลไปยังเซิร์ฟเวอร์ ──► [ อัปเดตข้อมูลลงชีตสำเร็จ ]
                  ▲                                                                 │
                  │                                                                 ▼
                  └────── เรียกคำสั่งโหลดข้อมูลซ้ำ (loadAllData) ───────────────────┘
                       (หน้าเว็บและหน้าจัดการอัปเดตข้อมูลแบบเรียลไทม์ทันทีโดยไม่ต้องรีเฟรชเบราว์เซอร์)
```

### 5.1 ระบบอัปเดตแบบลื่นไหลไร้รอยต่อ (Reactive Front-end Rendering)
*   **วิธีการทำงาน:** ห้ามเขียนคำสั่งให้ระบบกดรีเฟรชหน้าต่างเบราว์เซอร์ (`location.reload()`) เมื่อเกิดการเพิ่ม ลบ หรือแก้ไขข้อมูลใด ๆ
*   **ตรรกะควบคุม:** ในทุก ๆ การตอบรับผลทำงานสำเร็จจากหลังบ้าน (`withSuccessHandler`) ของคำสั่ง CRUD ให้เรียกฟังก์ชันดึงข้อมูลใหม่มาทับตัวแปรเดิมทันที (เรียกฟังก์ชันดึงข้อมูลหลักเช่น `loadAllData()`) วิธีนี้จะทำให้ตารางข้อมูลในหน้าแผงควบคุม แกลเลอรีภาพ และหน้าเพจต่าง ๆ อัปเดตเนื้อหาใหม่ล่าสุดให้ผู้ชมได้เห็นในทันทีแบบนุ่มนวล

### 5.2 ส่วนจัดการคณะผู้บริหารและบุคลากร (Board CRUD Dashboard)
*   **การแสดงผล:** แสดงทำเนียบผู้บริหารในหน้าแผงควบคุมเรียงตามตัวเลขหลัก `order`
*   **ปุ่มจัดการ:** มีปุ่มเพิ่มข้อมูลผู้บริหาร, ปุ่มแก้ไข และปุ่มกดลบ
*   **ฟอร์มโมดอล:**
    *   รองรับช่องกรอกชื่อ-นามสกุล, ตำแหน่ง, ตัวเลขจัดลำดับ (Order) และช่องใส่ลิงก์รูปภาพ
    *   มีปุ่มเลือกไฟล์ภาพเพื่ออัปโหลดเข้า Google Drive โดยตรง เมื่ออัปโหลดเสร็จให้ระบบนำที่อยู่ภาพมาใส่ในช่องข้อความอัตโนมัติพร้อมแสดงรูปภาพจำลองทันที

### 5.3 ส่วนจัดการภาพแบบหลายรูป (Multi-Image Uploader Grid)
*   ที่หน้าต่างโมดอลเพิ่ม/แก้ไขข่าวสารประชาสัมพันธ์และภาพกิจกรรม
*   **พฤติกรรมการทำงาน:** 
    *   แอดมินสามารถคลิกอัปโหลดรูปภาพทีละรูปเข้ามาผสมรวมกันได้หลายครั้ง โดยรูปที่อัปโหลดจะถูกส่งขึ้นระบบไดรฟ์ทีละรูป
    *   เมื่อหน้าหลังบ้านส่ง URL ลิงก์กลับมา ให้หน้าบ้านรวมเข้ากับค่าเดิมโดยแยกด้วยสัญลักษณ์จุลภาค (Comma)
    *   มีช่องสำหรับวางที่อยู่รูปภาพตรง ๆ (Manual URL Input) แล้วกดปุ่มเพิ่มเพื่อแนบรูปภาพเพิ่มเติมได้
    *   หน้าบ้านต้องแสดงภาพขนาดย่อ (Thumbnail Grid) ของทุกรูปที่ระบุไว้ในช่องข้อความเก็บ URL พร้อมมีปุ่มกากบาท `✕` ที่มุมบนขวาของแต่ละรูปเพื่อให้แอดมินสามารถคลิกลบเฉพาะรูปนั้น ๆ ออกจากอัลบั้มได้ในทันที

## 6. ระบบกล่องข้อความติดต่อและแจ้งเตือน (Contact Us & Admin Inbox Portal)

เพื่ออำนวยความสะดวกในการจัดเก็บ ค้นหา และแจ้งเตือนข้อความติดต่อสอบถามของผู้เยี่ยมชมเว็บไซต์โรงเรียน ระบบได้บูรณาการตารางข้อความติดต่อและการแจ้งเตือนอัตโนมัติเข้ามาดังนี้:

### 6.1 ตารางฐานข้อมูล `Messages`
เก็บข้อมูลแบบ row-by-row สำหรับข้อความติดต่อทุกรายการ:
*   `id` (UUID): รหัสอ้างอิงเฉพาะของแต่ละข้อความ (สร้างอัตโนมัติฝั่ง Server)
*   `name` (String): ชื่อ-นามสกุลของผู้ส่งข้อมูลติดต่อ
*   `email` (String): อีเมลผู้ส่งสำหรับการติดต่อกลับ
*   `subject` (String): หัวข้อเรื่อง/ประเภทเนื้อหาที่ติดต่อ
*   `message` (String): รายละเอียดเนื้อหาคำถามหรือข้อเสนอแนะเต็มรูปแบบ
*   `date` (ISO Date String): วันและเวลาจัดส่งตามเขตเวลาของระบบสคริปต์
*   `status` (String): สถานะการประมวลผลข้อความ: `unread` (ยังไม่ได้อ่าน - ค่าเริ่มต้น), `read` (เปิดอ่านแล้ว), `replied` (ตอบกลับทางอีเมลแล้ว)
*   `notes` (String): โน้ตภายในย่อสำหรับผู้ดูแลระบบจดบันทึกประวัติการติดต่อ/ข้อสังเกตส่วนตัว

### 6.2 โฟลว์การบันทึกข้อมูลและส่งอีเมลแจ้งเตือนผู้บริหาร (Message Submission Flow)
```
[ ผู้เยี่ยมชมส่งข้อความ ] 
       │ 
       ▼
[ Front-end: แสดงสถานะหมุนโหลด, ปิดการกดปุ่มซ้ำ ]
       │ 
       ▼ (ส่งคำขอ addContactMessage ไปหลังบ้าน)
[ Back-end: เพิ่มข้อมูลลง Messages Sheet (status: 'unread') ]
       │

       │            │
```

### 6.3 ระบบจัดการกล่องข้อความใน Admin Dashboard (Admin Inbox Management Portal)
ผู้ดูแลระบบที่ล็อกอินและมี `SESSION_TOKEN` ที่ถูกต้อง สามารถเข้าถึงหน้าบริหารจัดการ Inbox ได้ดังนี้:
1.  **Sidebar Badge Count:** 
    *   เมื่อหน้าหลัก (Overview) หรือหน้าแผงควบคุมโหลดขึ้นมา ระบบจะสแกนหาข้อความที่มีสถานะ `unread` ทั้งหมด และอัปเดตตัวเลขจำนวนข้อความค้างอ่านบนปุ่ม "กล่องข้อความติดต่อ" ในแถบเมนูด้านซ้ายแบบทันที เพื่อทำหน้าที่แจ้งเตือนจำนวนข้อความเข้าใหม่
2.  **การค้นหาและคัดกรองแบบไดนามิก (Dynamic Search & Filter):**
    *   มีช่องป้อนคำค้นหา (Search Box) ตรวจสอบคำค้นหาแบบไม่ระบุพิมพ์เล็ก-ใหญ่ (Case-Insensitive) โดยค้นหาแบบครอบคลุมทั้ง ชื่อผู้ส่ง, อีเมล, หัวข้อ, และตัวเนื้อหาข้อความติดต่อ
    *   มีช่อง dropdown เลือกตัวกรองสถานะ (Status Filter) เพื่อคัดแยกเฉพาะข้อความที่ต้องการดู เช่น ข้อความค้างอ่าน (`unread`), อ่านแล้ว (`read`), หรือตอบกลับเสร็จสิ้น (`replied`)
3.  **โมดอลแสดงรายละเอียดข้อความติดต่อ (Message Inspector Modal):**
    *   **การอัปเดตสถานะอัตโนมัติ (Auto-Mark as Read):** เมื่อผู้ดูแลระบบคลิกเลือกเปิดดูรายละเอียดข้อความใด ๆ ที่มีสถานะเดิมเป็น `unread` หน้าบ้านจะส่งคำขอไปยังหลังบ้านทันทีเพื่ออัปเดตสถานะเป็น `read` โดยอัตโนมัติ เพื่อประหยัดขั้นตอนการทำงานของแอดมิน และจำนวนตัวแจ้งเตือนที่แถบเมนูด้านข้างจะลดจำนวนลงทันที
    *   **บันทึกโน้ตภายใน (Admin Internal Notes):** แอดมินสามารถป้อนข้อความจดบันทึกภายใน และคลิกปุ่มบันทึกเพื่อจัดเก็บสถานะหรือประวัติลงสเปรดชีตฐานข้อมูลโดยตรง
    *   **การลบข้อความ (Contact Message Deletion):** มีปุ่มลบข้อความติดต่อ โดยระบบจะเรียกใช้โมดอลยืนยันแบบกำหนดเอง (Custom Confirm Modal) ป้องกันการเผลอกดผิดลบข้อมูล เมื่อยืนยันการลบแล้ว จะส่ง Token และ ID ไปทำการลบแถวใน Google Sheet ทันที ก่อนทำการดึงข้อมูลใหม่มาเรนเดอร์ตารางต้อนรับการเปลี่ยนแปลงแบบไร้รอยต่อ

## 7. ระบบโปรแกรมแก้ไขข้อความ WordPress-style Classic Editor (Rich Text Editor)

เพื่อเพิ่มความสามารถให้ผู้ดูแลระบบออกแบบการจัดวางเนื้อหาข่าวสารและข่าวประกาศอัปเดตได้อิสระ โดยไม่ต้องมีความรู้ด้านโค้ด HTML จึงได้พัฒนาโปรแกรมแก้ไขข้อความ Rich Text Editor ที่มีคุณสมบัติดังนี้:

### 7.1 โครงสร้างและการแสดงผลแถบเครื่องมือ (Editor Layout & Toolbar)
*   **Editor Container:** ทำการซ่อน Textarea ดั้งเดิม แล้วครอบพื้นที่ด้วย `.classic-editor-wrapper` ซึ่งประกอบไปด้วย แถบเครื่องมือด้านบน (`.classic-editor-toolbar`) และพื้นที่แก้ไขข้อมูลด้านล่าง (`.classic-editor-content`) ที่เปิดใช้งานคุณสมบัติ `contenteditable="true"` เพื่อให้พิมพ์และโต้ตอบได้เสมือนแอปเอกสารทั่วไป
*   **แถบเครื่องมือประกอบด้วย:**
    1.  **Format Block Select:** เมนู Dropdown สำหรับเลือกจัดรูปแบบหัวข้อ เช่น Normal text (`<p>`), Heading 2 (`<h2>`), Heading 3 (`<h3>`)
    2.  **ปุ่มจัดตัวหนา ตัวเอียง ขีดเส้นใต้:** ส่งคำสั่งผ่าน `document.execCommand('bold')`, `document.execCommand('italic')`, และ `document.execCommand('underline')`
    3.  **กล่องเลือกสีตัวอักษร (Color Picker):** แอดมินสามารถคลิกเลือกพาเลตสีเพื่อใช้แต่งสีข้อความส่วนที่ลากแถบคลุมไว้ได้ตามต้องการ โดยส่งค่าสีไปยังคำสั่ง `document.execCommand('foreColor')`
    4.  **ปุ่มแทรกลิงก์ (Insert Link):** เปิดกล่องโต้ตอบให้แอดมินใส่ URL ลิงก์ที่ต้องการแปลงข้อความที่คลุมดำให้กลายเป็นลิงก์คลิกได้
    5.  **ปุ่มสร้างรายการแบบมีหัวข้อย่อย (Lists):** รองรับทั้งรายการสัญลักษณ์วงกลม (`insertUnorderedList`) และรายการตัวเลขเรียงลำดับ (`insertOrderedList`)
    6.  **ปุ่มดูและแก้ไขโค้ดดิ้งดิบ (Toggle HTML Source Mode):** ปุ่ม `</>` สลับโหมดการแก้ไข เมื่อกดจะซ่อนหน้า Content Editable และแสดงหน้าต่าง Textarea โค้ดดิ้งดิบแทน เพื่อให้โปรแกรมเมอร์หรือแอดมินขั้นสูงปรับแก้โค้ด HTML ได้โดยตรง

### 7.2 ตรรกะการซิงค์ข้อมูล (Bi-directional Data Syncing)
*   เมื่อผู้ใช้งานทำการพิมพ์ ปรับแต่ง หรือสลับโหมดในตัวแก้ไข Rich Text Editor ตัวระบบจะนำเนื้อหา HTML ล่าสุดไปบันทึกลงในช่อง Textarea เดิมที่ซ่อนไว้ทันที พร้อมส่งสัญญาณเตือนอินพุต (`Event('input')`) ไปยังโค้ดส่วนอื่น ๆ
*   เมื่อระบบเปิดฟอร์มเพื่อแก้ไขข่าวสารหรือประกาศเก่า ระบบจะอ่านค่า HTML เดิมจาก Textarea แล้วส่งเข้าไปตั้งค่าแสดงผลเริ่มต้นที่พื้นที่ Content Editable เสมอ ป้องกันการทำงานไม่ตรงกันของข้อมูล (Data Desync)

## 8. ระบบปัญญาประดิษฐ์ผู้ช่วยแอดมิน (Google Gemini AI Assistant Integration)

ระบบเสริมปัญญาประดิษฐ์เพื่อช่วยให้ผู้ดูแลระบบเขียนบทความข่าวสาร คำอธิบายภาพ และสร้างภาพประกอบได้อย่างรวดเร็ว โดยมีรายละเอียดการทำงานดังนี้:

### 8.1 สิทธิ์การตั้งค่าและการเข้าถึง (AI Access Key Setup)
*   ผู้ดูแลระบบสามารถนำ **API Key** ที่ได้จาก Google AI Studio มาป้อนเก็บไว้ในหน้าตั้งค่าเว็บไซต์โรงเรียนภายใต้ตัวแปรคีย์ `geminiApiKey` (ระบบจัดเก็บแบบเข้ารหัสผ่านช่องอินพุตประเภท password)
*   คีย์จะถูกดึงมาตรวจสอบและเรียกใช้งานบน Server-side Apps Script เสมอ เพื่อความปลอดภัย ไม่ให้คีย์รั่วไหลไปสู่หน้าต่างเบราว์เซอร์ของคนทั่วไป

### 8.2 ตรรกะการเจนเนื้อหาอักษร (Text Generation Logic)
*   **การเชื่อมต่อ:** ใช้ API คอนเนคเตอร์ เรียกไปยัง Gemini API โมเดล `gemini-2.5-flash` ผ่านโปรโตคอล POST
*   **กระบวนการ:** เมื่อแอดมินป้อนคำสั่งหรือโครงร่างข่าวที่ป๊อปอัปฝั่งหน้าบ้าน และส่งคำสั่งเจน ตัวหลังบ้านจะส่งคำขอและรอรับคำตอบเนื้อหาบทความแบบเรียลไทม์
*   **การใช้งาน:** เมื่อแอดมินตรวจสอบผลลัพธ์ผ่านหน้า Preview และกด "นำข้อมูลไปใช้" ตัวระบบจะนำข้อความที่ได้แทนที่เข้าไปในช่องฟอร์ม พร้อมแปลงเครื่องหมายเว้นบรรทัด (`\n`) ให้กลายเป็นแท็ก HTML `<br>` เพื่อให้การขึ้นบรรทัดใหม่แสดงผลได้อย่างถูกต้องบนโปรแกรม Rich Text Editor

### 8.3 ตรรกะการสร้างรูปภาพและเก็บลงไดรฟ์โดยอัตโนมัติ (AI Image Generation & Drive Engine)
*   **การเชื่อมต่อ:** ใช้ Imagen API โมเดล `imagen-3.0-generate-002` ในการวาดและส่งภาพตอบกลับในรูปแบบ Base64
*   **กระบวนการอัปโหลดอัตโนมัติ (Auto-Upload Pipeline):**
    1.  เมื่อหลังบ้านได้รับข้อมูลรูปภาพ Base64 จาก Imagen API ตัวเซิร์ฟเวอร์จะทำการถอดรหัส Base64
    2.  เซิร์ฟเวอร์จะเรียกใช้ฟังก์ชัน `uploadImage()` เดิมเพื่อนำไฟล์ภาพดังกล่าวไปเก็บเข้าโฟลเดอร์ Google Drive ของโรงเรียนให้ทันทีโดยไม่มีการดาวน์โหลดมาลงเครื่อง
    3.  เมื่อบันทึกภาพสำเร็จ ไดรฟ์จะส่งลิงก์ภาพขนาดย่อที่มีประสิทธิภาพ (`w1200 thumbnail url`) กลับคืนมาเป็นผลลัพธ์
    4.  หน้าบ้านจะนำ URL ภาพดังกล่าว ไปใส่ต่อท้ายในช่องรายการเก็บรูปภาพของแบบฟอร์มนั้น ๆ (คั่นด้วยจุลภาค) พร้อมเรนเดอร์ภาพตัวอย่างขนาดย่อขึ้นที่หน้าจอแอดมินทันที เพื่อให้สามารถใช้ปุ่มกากบาท `✕` ลบหรือสลับภาพประกอบข่าวสารได้ทันที
